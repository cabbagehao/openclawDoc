## 問題

現在、コマンド固有のチェック (`/new` や `/reset` など) が用意されています。
ホワイトリストが空の場合でも、ACP バインドされたチャネル/トピック内で作業する必要があります。
これにより、当面の UX の問題は解決されますが、コマンド名ベースの例外は拡張できません。

## 長期的な形状

コマンド認可をアドホック ハンドラー ロジックからコマンド メタデータに移動し、
共有ポリシー評価者。

### 1) 認証ポリシーのメタデータをコマンド定義に追加します

各コマンド定義では、認証ポリシーを宣言する必要があります。形状の例:

```ts
type CommandAuthPolicy =
  | { mode: "owner_or_allowlist" } // default, current strict behavior
  | { mode: "bound_acp_or_owner_or_allowlist" } // allow in explicitly bound ACP conversations
  | { mode: "owner_only" };
```

`/new` および `/reset` は `bound_acp_or_owner_or_allowlist` を使用します。
他のほとんどのコマンドは `owner_or_allowlist` のままになります。

### 2) チャネル間で 1 人の評価者を共有する

以下を使用してコマンド認証を評価するヘルパーを 1 つ導入します。

- コマンドポリシーメタデータ
- 送信者の認証状態
- 解決された会話バインディング状態

これを避けるために、Telegram と Discord の両方のネイティブ ハンドラーは同じヘルパーを呼び出す必要があります。
動作のドリフト。

### 3) バイパス境界として binding-match を使用するポリシーでバインドされた ACP バイパスが許可されている場合、バインディングが設定されている場合にのみ許可します

現在の会話の一致が解決されました (現在の会話の理由だけではありません)

これにより、境界が明確に保たれ、偶発的な拡大が最小限に抑えられます。

## これが優れている理由

- コマンド名の条件を追加せずに、将来のコマンドに合わせて拡張します。
- チャネル全体で動作の一貫性を保ちます。
- 明示的なバインディングの一致を要求することにより、現在のセキュリティ モデルを維持します。
- ホワイトリストを普遍的な要件ではなく、オプションの強化のままにします。

## 展開計画 (将来)

## 非目標

- この提案は、ACP セッションのライフサイクル動作を変更しません。
- この提案では、すべての ACP バインド コマンドのホワイトリストは必要ありません。
- この提案は、既存のルート バインディング セマンティクスを変更しません。
