# Discord 非同期インバウンド ワーカー プラン

## 目的

インバウンドの Discord を非同期にすることで、ユーザー側の障害モードとして Discord リスナーのタイムアウトを削除します。

1. ゲートウェイ リスナーは受信イベントを迅速に受け入れて正規化します。
2. Discord 実行キューには、現在使用されているのと同じ順序境界をキーとするシリアル化されたジョブが保存されます。
3. ワーカーは、Carbon リスナーの有効期間外で実際のエージェントを実行します。
4. 実行が完了すると、応答は元のチャネルまたはスレッドに返されます。

これは、エージェントの実行自体がまだ進行中であるにもかかわらず、キューに入れられた Discord の実行が `channels.discord.eventQueue.listenerTimeout` でタイムアウトになる場合の長期的な修正です。

## 現在のステータス

この計画は部分的に実施されています。

すでに完了しています:

- Discord リスナーのタイムアウトと Discord の実行タイムアウトは別個の設定になりました。
- 受け入れられた受信 Discord ターンは `src/discord/monitor/inbound-worker.ts` にエンキューされます。
- ワーカーは、Carbon リスナーの代わりに長期実行ターンを所有するようになりました。
- 既存のルートごとの順序付けは、キュー キーによって保存されます。
- Discord ワーカー パスにはタイムアウト回帰カバレッジが存在します。

これが意味するところは、分かりやすく言えば次のとおりです。

- 本番タイムアウトのバグが修正されました
- Discord リスナーの予算が期限切れになったからといって、長期にわたるターンが終了することはなくなりました
- ワーカー アーキテクチャはまだ完成していません

まだ足りないもの:- `DiscordInboundJob` はまだ部分的にのみ正規化されており、ライブ ランタイム参照を保持しています。

- コマンド セマンティクス (`stop`、`new`、`reset`、将来のセッション制御) はまだ完全にはワーカーネイティブではありません
- 作業者の観察可能性とオペレータのステータスは依然として最小限です
- 再起動耐久性はまだありません

## これが存在する理由

現在の動作は、エージェントの完全なターンをリスナーの存続期間に結び付けます。

- `src/discord/monitor/listeners.ts` はタイムアウトと中止の境界を適用します。
- `src/discord/monitor/message-handler.ts` は、キューに入れられた実行をその境界内に保持します。
- `src/discord/monitor/message-handler.process.ts` は、メディアのロード、ルーティング、ディスパッチ、入力、ドラフト ストリーミング、および最終応答の配信をインラインで実行します。

このアーキテクチャには 2 つの悪い特性があります。

- 長くても健全なターンは、リスナーウォッチドッグによって中止される可能性があります
- ダウンストリームのランタイムが応答を生成した場合でも、ユーザーには応答が表示されません。

タイムアウトを増やすと効果はありますが、障害モードは変わりません。

## 非目標

- このパスでは、Discord 以外のチャンネルを再設計しないでください。
- 最初の実装では、これを汎用の全チャネル ワーカー フレームワークに拡張しないでください。
- 共有クロスチャネル インバウンド ワーカー抽象化はまだ抽出しないでください。重複が明らかな場合にのみ、低レベルのプリミティブを共有します。
- 安全に着陸するために必要な場合を除き、最初のパスで耐久性のあるクラッシュ リカバリーを追加しないでください。
- この計画では、ルート選択、バインディング セマンティクス、または ACP ポリシーを変更しないでください。

## 現在の制約現在の Discord 処理パスは依然として、長期的なジョブ ペイロード内に留まるべきではないいくつかのライブ ランタイム オブジェクトに依存しています

- カーボン `Client`
- 未加工の Discord イベントシェイプ
- メモリ内のギルド歴史マップ
- スレッド バインディング マネージャーのコールバック
- ライブタイピングとドラフトストリーム状態

すでに実行をワーカー キューに移動しましたが、正規化境界はまだ不完全です。現時点では、ワーカーは「同じライブ オブジェクトの一部を使用して同じプロセス内で後で実行」されており、完全にデータのみのジョブ境界ではありません。

## ターゲット アーキテクチャ

### 1. リスナー段階

`DiscordMessageListener` は入口ポイントのままですが、そのジョブは次のようになります。

- プリフライトおよびポリシーチェックを実行します
- 受け入れられた入力をシリアル化可能な `DiscordInboundJob` に正規化します
- ジョブをセッションごとまたはチャネルごとの非同期キューにエンキューします
- エンキューが成功すると、すぐに Carbon に戻ります

リスナーはエンドツーエンドの LLM ターンのライフタイムを所有すべきではなくなります。

### 2. 正規化されたジョブのペイロード

後でターンを実行するために必要なデータのみを含むシリアル化可能なジョブ記述子を導入します。

最小形状:- ルートアイデンティティ

- `agentId`
- `sessionKey`
- `accountId`
- `channel`
- 配送ID
  - 宛先チャネルID
  - 返信対象のメッセージID
  - スレッド ID が存在する場合
- 送信者の身元情報
  - 送信者ID、ラベル、ユーザー名、タグ
- チャネルコンテキスト
  - ギルドID
  - チャンネル名またはスラッグ
  - スレッドのメタデータ
  - システムプロンプトのオーバーライドを解決しました
- 正規化されたメッセージ本文
  - ベーステキスト
  - 効果的なメッセージテキスト
  - 添付ファイル記述子または解決されたメディア参照
- ゲート決定
  - 要件の結果について言及する
  - コマンド認可結果
  - バインドされたセッションまたはエージェントのメタデータ (該当する場合)

ジョブ ペイロードには、ライブ Carbon オブジェクトや可変クロージャが含まれていてはなりません。

現在の実装状況:

- 部分的に完了しました
- `src/discord/monitor/inbound-job.ts` が存在し、ワーカーのハンドオフを定義します
- ペイロードにはまだライブの Discord ランタイム コンテキストが含まれているため、さらに削減する必要があります

### 3. ワーカーステージ

以下を担当する Discord 固有のワーカー ランナーを追加します。

- `DiscordInboundJob` からターン コンテキストを再構築する
- 実行に必要なメディアと追加のチャンネルメタデータの読み込み
- エージェントの順番を派遣する
- 最終応答ペイロードの配信
- ステータスと診断の更新

推奨される場所:

- `src/discord/monitor/inbound-worker.ts`
- `src/discord/monitor/inbound-job.ts`

### 4. 注文モデル

特定のルート境界については、順序が現在と同等のままである必要があります。

推奨キー:- `resolveDiscordRunQueueKey(...)` と同じキュー キー ロジックを使用します。

これにより、既存の動作が維持されます。

- 1 つのバインドされたエージェントの会話はそれ自体とインターリーブされません
- 異なる Discord チャンネルは引き続き独立して進行できます

### 5. タイムアウトモデル

カットオーバー後は、2 つの別個のタイムアウト クラスがあります。

- リスナーのタイムアウト
  - 正規化とエンキューのみをカバーします
  - 短くする必要があります
- 実行タイムアウト
  - オプション、ワーカー所有、明示的、ユーザー表示
  - Carbon リスナー設定から誤って継承しないでください。

これにより、「Discord ゲートウェイ リスナーは生存している」と「エージェントの実行は正常である」との間の現在の偶発的な結合が解消されます。

## 推奨される実装フェーズ

### フェーズ 1: 正規化境界

- ステータス: 部分的に実装
- 完了:
  - `buildDiscordInboundJob(...)` を抽出しました
  - ワーカーハンドオフテストを追加しました
- 残り:
  - `DiscordInboundJob` プレーンデータのみにする
  - ライブランタイム依存関係をジョブごとのペイロードではなくワーカー所有のサービスに移動します
  - ライブリスナー参照をジョブにつなぎ戻すことで、プロセスコンテキストの再構築を停止します。

### フェーズ 2: インメモリ ワーカー キュー

- ステータス: 実装済み
- 完了:
  - 解決された実行キューキーをキーとする `DiscordInboundWorkerQueue` を追加しました
  - リスナーは、`processDiscordMessage(...)` を直接待つのではなく、ジョブをキューに入れます。
  - ワーカーはメモリ内のみでジョブをインプロセスで実行します

これは最初の機能カットオーバーです。

### フェーズ 3: プロセスの分割- ステータス: 未開始

- 配信、入力、およびドラフト ストリーミングの所有権をワーカー側のアダプターの背後に移動します。
- ライブ プリフライト コンテキストの直接使用をワーカー コンテキストの再構築に置き換えます。
- 必要に応じて `processDiscordMessage(...)` をファサードとして一時的に保持し、その後分割します。

### フェーズ 4: コマンド セマンティクス

- ステータス: 未開始
  作業がキューに入れられている場合でも、ネイティブの Discord コマンドが正しく動作することを確認します。

- `stop`
- `new`
- `reset`
- 今後のセッション制御コマンド

ワーカー キューは、アクティブなターンまたはキューに入れられたターンを対象とするコマンドに対して十分な実行状態を公開する必要があります。

### フェーズ 5: 可観測性とオペレーターの UX

- ステータス: 未開始
- キューの深さとアクティブなワーカー数を監視ステータスに出力します
- エンキュー時間、開始時間、終了時間、タイムアウトまたはキャンセルの理由を記録します。
- ワーカーが所有するタイムアウトまたは配信失敗をログに明確に記録する

### フェーズ 6: オプションの耐久性フォローアップ

- ステータス: 未開始
  インメモリ バージョンが安定した後でのみ:

- キューに入れられた Discord ジョブがゲートウェイの再起動を存続させるかどうかを決定します
- 「はい」の場合、ジョブ記述子と配信チェックポイントを永続化します。
- 「いいえ」の場合、明示的なメモリ内境界を文書化します。

着陸するために再起動回復が必要でない限り、これは別のフォローアップである必要があります。

## 今すぐ次のステップへ

次のステップは、ワーカー境界を部分的ではなく実際のものにすることです。

次にこれを実行します。

1. ライブ ランタイムの依存関係を `DiscordInboundJob` から移動します。
2. 代わりに、それらの依存関係を Discord ワーカー インスタンスに保持します。
3. キューに入れられたジョブを単純な Discord 固有のデータに削減します。
   - ルートアイデンティティ
   - 配信対象
   - 送信者情報
   - 正規化されたメッセージのスナップショット
   - ゲートとバインディングの決定
4. ワーカー内のプレーンデータからワーカー実行コンテキストを再構築します。

実際には、それは次のことを意味します。

- `client`
- `threadBindings`
- `guildHistories`
- `discordRestFetch`
- その他の変更可能なランタイム専用ハンドル

キューに入れられた各ジョブでの生存を停止し、代わりにワーカー自体またはワーカー所有のアダプターの背後で生存する必要があります。

それが完了したら、次のフォローアップは、`stop`、`new`、および `reset` のコマンド状態のクリーンアップです。

## テスト計画

既存のタイムアウト再現カバレッジを次の場所に保持します。

- `src/discord/monitor/message-handler.queue.test.ts`

次の新しいテストを追加します。1. リスナーはエンキュー後に完全なターンを待たずに戻ります 2. ルートごとの順序は保持されます 3. 異なるチャネルが同時に実行されている 4. 返信は元のメッセージの宛先に配信されます。5. `stop` は、ワーカーが所有するアクティブな実行をキャンセルします。6. ワーカーに障害が発生しても、後のジョブをブロックすることなく、目に見える診断が生成されます 7. ACP にバインドされた Discord チャネルは、ワーカーの実行下でも引き続き正しくルーティングされます。

## リスクと緩和策

- リスク: コマンド セマンティクスが現在の同期動作から逸脱する
  緩和策: 後でではなく、同じカットオーバーでコマンド状態の配管を配置します。

- リスク: 返信配信によりスレッドまたは返信先コンテキストが失われる
  軽減策: `DiscordInboundJob` で配信 ID をファーストクラスにする

- リスク: 再試行またはキューの再起動中に送信が重複する
  軽減策: 最初のパスをメモリ内のみに保持するか、永続化の前に明示的な配信冪等性を追加します。

- リスク: `message-handler.process.ts` は、移行中に推論することが難しくなります
  緩和: ワーカーのカットオーバー前またはカットオーバー中に、正規化、実行、および配信ヘルパーに分割されます。

## 受け入れ基準

計画は次の時点で完了します。1. Discord リスナーのタイムアウトにより、健全な長時間実行ターンが中止されなくなりました。2. リスナーの有効期間とエージェント ターンの有効期間は、コード内の別個の概念です。3. 既存のセッションごとの順序は保持されます。4. ACP にバインドされた Discord チャネルは、同じワーカー パスを通じて動作します。5. `stop` は、古いリスナー所有のコール スタックではなく、ワーカー所有の実行をターゲットにします。6. タイムアウトと配信の失敗は、サイレント リスナーのドロップではなく、明示的なワーカーの結果になります。

## 残りの着陸戦略

フォローアップ PR でこれを完了します。

1. `DiscordInboundJob` をプレーンデータのみにし、ライブランタイム参照をワーカーに移動します
2. `stop`、`new`、および `reset` のコマンド状態の所有権をクリーンアップします。
3. ワーカーの可観測性とオペレーターのステータスを追加する
4. 耐久性が必要かどうかを判断するか、メモリ内の境界を明示的に文書化する

Discord のみを維持し、時期尚早なクロスチャネル ワーカーの抽象化を回避し続ける場合、これは依然として限定的なフォローアップです。
