# ブラウザによる CDP リファクタリング計画の評価

## コンテキスト

`act:evaluate` は、ページ内でユーザーが指定した JavaScript を実行します。現在は Playwright 経由で実行されます
(`page.evaluate` または `locator.evaluate`)。 Playwright は CDP コマンドをページごとにシリアル化するため、
評価がスタックしたり、長時間実行されたりすると、ページ コマンド キューがブロックされ、その後のすべてのアクションが実行される可能性があります
そのタブでは「スタック」しているように見えます。

PR #13498 は、実用的なセーフティ ネット (有界評価、伝播中止、ベストエフォート) を追加します。
回復）。このドキュメントでは、`act:evaluate` を本質的に作成する大規模なリファクタリングについて説明します。
Playwright から分離されているため、スタックした評価では通常の Playwright 操作を中断できません。

## 目標

- `act:evaluate` は、同じタブでのその後のブラウザーのアクションを永続的にブロックすることはできません。
- タイムアウトはエンドツーエンドの信頼できる唯一の情報源であるため、呼び出し元はバジェットに依存できます。
- アボートとタイムアウトは、HTTP とインプロセス ディスパッチで同じように扱われます。
- Playwright をすべてオフにすることなく、評価の対象となる要素がサポートされます。
- 既存の呼び出し元とペイロードに対する下位互換性を維持します。

## 非目標- すべてのブラウザーのアクション (クリック、入力、待機など) を CDP 実装に置き換えます

- PR #13498 で導入された既存のセーフティ ネットを削除します (有用なフォールバックのままです)。
- 既存の `browser.evaluateEnabled` ゲートを超える新しい安全でない機能を導入します。
- 評価用にプロセス分離 (ワーカー プロセス/スレッド) を追加します。まだ回復が難しいと思われる場合
  このリファクタリング後のスタック状態、これはフォローアップのアイデアです。

## 現在のアーキテクチャ (行き詰まる理由)

高いレベルで:

- 呼び出し元は `act:evaluate` をブラウザー制御サービスに送信します。
- ルート ハンドラーは Playwright を呼び出して JavaScript を実行します。
- Playwright はページ コマンドをシリアル化するため、終了しない評価はキューをブロックします。
- キューがスタックしているということは、その後のタブ上でのクリック/入力/待機操作がハングしているように見える可能性があることを意味します。

## 提案されたアーキテクチャ

### 1. 期限の伝達

単一の予算概念を導入し、そこからすべてを導き出します。

- 呼び出し元は `timeoutMs` (または将来の期限) を設定します。
- 外部リクエストのタイムアウト、ルート ハンドラー ロジック、ページ内の実行バジェット
  すべてが同じバジェットを使用し、シリアル化オーバーヘッドが必要な場合には小さなヘッドルームを備えています。
- 中止はどこにでも `AbortSignal` として伝播されるため、キャンセルは一貫しています。

実装の方向性:- 以下を返す小さなヘルパー (`createBudget({ timeoutMs, signal })` など) を追加します。

- `signal`: リンクされた AbortSignal
- `deadlineAtMs`: 絶対的な期限
- `remainingMs()`: 子操作の残りの予算
- このヘルパーは次の場合に使用します。
  - `src/browser/client-fetch.ts` (HTTP およびインプロセス ディスパッチ)
  - `src/node-host/runner.ts` (プロキシ パス)
  - ブラウザーアクションの実装 (Playwright および CDP)

### 2. 個別の評価エンジン (CDP パス)

Playwright のページごとのコマンドを共有しない CDP ベースの評価実装を追加します。
行列。重要なプロパティは、評価トランスポートが別個の WebSocket 接続であることです。
ターゲットに接続された別の CDP セッション。

実装の方向性:

- 新しいモジュール (`src/browser/cdp-evaluate.ts` など):
  - 構成された CDP エンドポイント (ブラウザー レベルのソケット) に接続します。
  - `Target.attachToTarget({ targetId, flatten: true })` を使用して `sessionId` を取得します。
  - 次のいずれかを実行します。
    - `Runtime.evaluate` ページ レベルの評価、または
    - 要素評価用の `DOM.resolveNode` と `Runtime.callFunctionOn`。
  - タイムアウトまたは中止時:
    - `Runtime.terminateExecution` セッションのベストエフォートを送信します。
    - WebSocket を閉じて、明確なエラーを返します。

注:

- これでもページ内で JavaScript が実行されるため、終了すると副作用が生じる可能性があります。勝利
  それは、Playwright のキューに割り込まず、トランスポート時にキャンセルできることです。
  CDP セッションを強制終了してレイヤーを削除します。### 3. 参照ストーリー (完全な書き換えを行わない要素ターゲティング)

難しいのは要素のターゲティングです。 CDP には DOM ハンドルまたは `backendDOMNodeId` が必要ですが、
現在、ほとんどのブラウザー アクションは、スナップショットからの参照に基づいた Playwright ロケーターを使用します。

推奨されるアプローチ: 既存の参照を保持しますが、オプションの CDP 解決可能な ID をアタッチします。

#### 3.1 保存された参照情報の拡張

保存されたロール参照メタデータを拡張して、必要に応じて CDP ID を含めます。

- 今日: `{ role, name, nth }`
- 提案: `{ role, name, nth, backendDOMNodeId?: number }`

これにより、既存のすべての Playwright ベースのアクションが機能し続け、CDP 評価が受け入れられるようになります。
`backendDOMNodeId` が使用可能な場合は、同じ `ref` 値。

#### 3.2 スナップショット時に backendDOMNodeId を設定する

ロールのスナップショットを作成する場合:

1. 既存のロール参照マップを今日 (ロール、名前、n 番目) として生成します。
2. CDP (`Accessibility.getFullAXTree`) 経由で AX ツリーをフェッチし、次の並列マップを計算します。
   `(role, name, nth) -> backendDOMNodeId` 同じ重複処理ルールを使用します。
3. 現在のタブに保存されている参照情報に ID をマージして戻します。

参照のマッピングが失敗した場合は、`backendDOMNodeId` を未定義のままにしてください。これにより、この機能が実現されます
ベストエフォート型で安全に展開できます。

#### 3.3 Ref を使用して動作を評価する

`act:evaluate`:

- `ref` が存在し、`backendDOMNodeId` がある場合、CDP を介して要素を評価します。
- `ref` は存在するが、`backendDOMNodeId` がない場合は、Playwright パスにフォールバックします (
  セーフティネット）。オプションの避難ハッチ:

- 高度な呼び出し元に対して `backendDOMNodeId` を直接受け入れるようにリクエスト シェイプを拡張します (および
  デバッグ用)、`ref` をプライマリ インターフェイスとして維持します。

### 4. 最後の手段である回復手段を確保しておく

CDP 評価を使用する場合でも、タブまたは接続をウェッジする他の方法があります。保管してください
最後の手段としての既存の回復メカニズム (実行の終了 + Playwright の切断)
ため:

- 従来の呼び出し元
- CDP アタッチがブロックされている環境
- 劇作家の予期せぬエッジケース

## 実装計画 (単一反復)

### 成果物

- Playwright のページごとのコマンド キューの外で実行される CDP ベースの評価エンジン。
- 呼び出し側とハンドラーによって一貫して使用される、単一のエンドツーエンドのタイムアウト/中止バジェット。
- 要素評価のためにオプションで `backendDOMNodeId` を運ぶことができる Ref メタデータ。
- `act:evaluate` は、可能な場合は CDP エンジンを優先し、そうでない場合は Playwright にフォールバックします。
- スタックした評価が後のアクションに影響を与えないことを証明するテスト。
- 失敗とフォールバックを可視化するログ/メトリクス。

### 実装チェックリスト1. 共有「予算」ヘルパーを追加して、`timeoutMs` + アップストリーム `AbortSignal` を次のようにリンクします

- 単一の `AbortSignal`
- 絶対的な期限
- ダウンストリーム操作用の `remainingMs()` ヘルパー

2. `timeoutMs` がどこでも同じことを意味するように、そのヘルパーを使用するようにすべての呼び出し元パスを更新します。
   - `src/browser/client-fetch.ts` (HTTP およびインプロセス ディスパッチ)
   - `src/node-host/runner.ts` (ノードプロキシパス)
   - `/act` を呼び出す CLI ラッパー (`--timeout-ms` を `browser evaluate` に追加)
3. `src/browser/cdp-evaluate.ts` を実装します。
   - ブラウザレベルの CDP ソケットに接続します
   - `Target.attachToTarget` で `sessionId` を取得します
   - ページ評価のために `Runtime.evaluate` を実行します
   - 要素評価のために `DOM.resolveNode` + `Runtime.callFunctionOn` を実行します
   - タイムアウト/中止時: ベストエフォート `Runtime.terminateExecution` その後、ソケットを閉じます
4. 保存されたロール参照メタデータを拡張して、オプションで `backendDOMNodeId` を含めます。
   - Playwright アクションの既存の `{ role, name, nth }` 動作を維持する
   - CDP 要素ターゲティングに `backendDOMNodeId?: number` を追加
5. スナップショットの作成中に `backendDOMNodeId` を設定します (ベストエフォート):
   - CDP 経由で AX ツリーを取得 (`Accessibility.getFullAXTree`)
   - `(role, name, nth) -> backendDOMNodeId` を計算し、保存された参照マップにマージします
   - マッピングがあいまいまたは欠落している場合は、ID を未定義のままにしておきます
6. `act:evaluate` ルーティングを更新します。
   - `ref` がない場合: 常に CDP 評価を使用します- `ref` が `backendDOMNodeId` に解決される場合: CDP 要素の評価を使用します
   - それ以外の場合: Playwright の評価にフォールバックします (まだ制限されており、中止可能です)
7. デフォルトのパスではなく、既存の「最後の手段」回復パスをフォールバックとして保持します。
8. テストを追加します。
   - スタックした評価が予算内でタイムアウトになり、次のクリック/タイプが成功する
   - 中止は評価をキャンセルし (クライアントの切断またはタイムアウト)、後続のアクションのブロックを解除します。
   - マッピングの失敗は Playwright にきれいにフォールバックします
9. 可観測性を追加します。
   - 期間とタイムアウトのカウンターを評価する
   - terminateExecution の使用法
   - フォールバック率 (CDP -> Playwright) とその理由

### 合格基準

- 意図的にハングされた `act:evaluate` は呼び出し元のバジェット内で戻り、
  タブをクリックして後のアクションを実行します。
- `timeoutMs` は、CLI、エージェント ツール、ノード プロキシ、およびインプロセス呼び出し全体で一貫して動作します。
- `ref` を `backendDOMNodeId` にマッピングできる場合、要素の評価は CDP を使用します。それ以外の場合は
  フォールバック パスはまだ制限されており、回復可能です。

## テスト計画- 単体テスト

- `(role, name, nth)` ロール参照と AX ツリー ノード間のロジックの一致。
- 予算ヘルパーの動作 (ヘッドルーム、残り時間の計算)。
- 統合テスト:
  - CDP はタイムアウトの戻り値を予算内で評価し、次のアクションをブロックしません。
  - 中止は評価をキャンセルし、ベストエフォートによる終了をトリガーします。
- 契約テスト:
  - `BrowserActRequest` と `BrowserActResponse` の互換性が維持されていることを確認します。

## リスクと緩和策

- マッピングが不完全です:
  - 軽減策: ベストエフォート型マッピング、Playwright 評価へのフォールバック、およびデバッグ ツールの追加。
- `Runtime.terminateExecution` には副作用があります:
  - 軽減策: タイムアウト/中止の場合にのみ使用し、エラー時の動作を文書化します。
- 追加のオーバーヘッド:
  - 軽減策: スナップショットが要求されたときにのみ AX ツリーをフェッチし、ターゲットごとにキャッシュして保持します。
    CDP セッションは短命でした。
- 拡張リレーの制限:
  - 軽減策: ページごとのソケットが利用できない場合は、ブラウザ レベルの接続 API を使用します。
    現在の Playwright パスをフォールバックとして保持します。
