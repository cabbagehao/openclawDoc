# Workspace Memory v2 (オフライン): 研究ノート

ターゲット: Clawd スタイルのワークスペース (`agents.defaults.workspace`、デフォルト `~/.openclaw/workspace`)。「メモリ」は 1 日あたり 1 つの Markdown ファイル (`memory/YYYY-MM-DD.md`) と、安定したファイルの小さなセット (例: `memory.md`、`SOUL.md`) として保存されます。

このドキュメントでは、Markdown を正規のレビュー可能な真実の情報源として保ちながら、派生インデックスを介して **構造化再現** (検索、エンティティの概要、信頼度の更新) を追加する **オフライン ファースト** メモリ アーキテクチャを提案します。

## なぜ変更するのでしょうか?

現在の設定 (1 日あたり 1 ファイル) は次の場合に最適です。

- 「追加専用」ジャーナリング
- 人間による編集
- git に裏付けられた耐久性 + 監査可能性
- 低摩擦キャプチャ (「書き留めるだけ」)

以下の場合に弱いです。

- 高再現率検索 (「X について何を決めましたか?」、「最後に Y を試したのはいつですか?」)
- 多くのファイルを再読することなく、エンティティ中心の回答 (「アリス / キャッスル / ウェアレイについて教えてください」)
- 意見/好みの安定性 (およびそれが変化したときの証拠)
- 時間的制約 (「2025 年 11 月に何が真実でしたか?」) と紛争解決

## 設計目標- **オフライン**: ネットワークなしで動作します。ラップトップ/城で実行できます。クラウドへの依存はありません

- **説明可能**: 取得されたアイテムは帰属可能 (ファイル + 場所) であり、推論から分離できる必要があります。
- **儀式が少ない**: 毎日のログ記録はマークダウンのままで、重いスキーマ作業はありません。
- **増分**: v1 は FTS でのみ役立ちます。セマンティック/ベクターとグラフはオプションのアップグレードです。
- **エージェントに優しい**: 「トークン予算内でのリコール」が簡単になります (小さなファクトの束を返す)。

## 北極星モデル (ハインドサイト × レッタ)

ブレンドする 2 つの部分:

1. **Letta/MemGPT スタイルの制御ループ**

- 小さな「核心」を常にコンテキスト内に保持します (ペルソナ + 重要なユーザー事実)
- それ以外はすべてアウトオブコンテキストであり、ツールを介して取得されます
- メモリ書き込みは明示的なツール呼び出し (追加/置換/挿入) であり、永続化され、次のターンに再挿入されます。

2. **後知恵スタイルのメモリ基板**

- 観察されたもの、信じられているもの、要約されたものを区別する
- 保持/リコール/反映をサポート
- 証拠とともに進化できる自信に満ちた意見
- エンティティ認識検索 + 一時クエリ (完全なナレッジ グラフがなくても)

## 提案されたアーキテクチャ (Markdown の真実の情報源 + 派生インデックス)

### Canonical ストア (git フレンドリー)

`~/.openclaw/workspace` を人間が判読できる正規のメモリとして保持します。

推奨されるワークスペースのレイアウト:

```
~/.openclaw/workspace/
  memory.md                    # small: durable facts + preferences (core-ish)
  memory/
    YYYY-MM-DD.md              # daily log (append; narrative)
  bank/                        # “typed” memory pages (stable, reviewable)
    world.md                   # objective facts about the world
    experience.md              # what the agent did (first-person)
    opinions.md                # subjective prefs/judgments + confidence + evidence pointers
    entities/
      Peter.md
      The-Castle.md
      warelay.md
      ...
```

注:- **日次ログは日次ログのままです**。 JSON に変換する必要はありません。

- `bank/` ファイルは **厳選**され、リフレクション ジョブによって生成され、手動で編集することができます。
- `memory.md` は「小さい + コアっぽい」ままです。Clawd にセッションごとに見てもらいたいもの。

### 派生ストア (マシンのリコール)

ワークスペースの下に派生インデックスを追加します (Git Tracked である必要はありません)。

```
~/.openclaw/workspace/.memory/index.sqlite
```

次のようにして戻します。

- ファクト + エンティティ リンク + 意見メタデータの SQLite スキーマ
- 字句呼び出し用の SQLite **FTS5** (高速、小型、オフライン)
- セマンティックリコールのためのオプションの埋め込みテーブル (まだオフライン)

インデックスは常に **Markdown から再構築可能**です。

## 保持 / 呼び出し / 反映 (操作ループ)

### 保持: 毎日のログを「事実」に正規化します

ここで重要となる Hindsight の重要な洞察: 小さな断片ではなく、**物語的で自己完結した事実**を保存することです。

`memory/YYYY-MM-DD.md` の実際的なルール:

- 一日の終わり (または途中) に、次の 2 ～ 5 個の箇条書きを含む `## Retain` セクションを追加します。
  - ナラティブ (クロスターンのコンテキストが保持される)
  - 自己完結型 (スタンドアロンは後で意味を持ちます)
  - タイプ + エンティティの言及でタグ付け

例:

```
## Retain
- W @Peter: Currently in Marrakech (Nov 27–Dec 1, 2025) for Andy’s birthday.
- B @warelay: I fixed the Baileys WS crash by wrapping connection.update handlers in try/catch (see memory/2025-11-27.md).
- O(c=0.95) @Peter: Prefers concise replies (&lt;1500 chars) on WhatsApp; long content goes into files.
```

最小限の解析:- タイプ接頭辞: `W` (世界)、`B` (経験/伝記)、`O` (意見)、`S` (観察/概要、通常生成)

- エンティティ: `@Peter`、`@warelay` など (スラッグは `bank/entities/*.md` にマップされます)
- 意見の信頼度: `O(c=0.0..1.0)` オプション

作成者にそれについて考えてほしくない場合: 反映ジョブはログの残りの部分からこれらの箇条書きを推測できますが、明示的な `## Retain` セクションを設けることが最も簡単な「品質レバー」です。

### 思い出してください: 派生インデックスに対するクエリ

リコールは以下をサポートする必要があります:

- **語彙**: 「正確な用語/名前/コマンドを検索する」(FTS5)
- **エンティティ**: 「X について教えてください」 (エンティティ ページ + エンティティにリンクされたファクト)
- **時間的**: 「11 月 27 日前後に起こったこと」 / 「先週以来」
- **意見**: 「ピーターは何が好きですか?」 (自信と証拠を持って)

返信形式はエージェントが使いやすいものにし、出典を引用する必要があります。

- `kind` (`world|experience|opinion|observation`)
- `timestamp` (ソース日、または存在する場合は抽出された時間範囲)
- `entities` (`["Peter","warelay"]`)
- `content` (物語の事実)
- `source` (`memory/2025-11-27.md#L12` など)

### 反映: 安定したページを作成し、信念を更新する

リフレクションは、次のようなスケジュールされたジョブ (毎日またはハートビート `ultrathink`) です。- 最近の事実 (エンティティの概要) から `bank/entities/*.md` を更新します

- 強化/矛盾に基づいて `bank/opinions.md` の信頼度を更新します
- オプションで `memory.md` への編集を提案します (「コアっぽい」永続的な事実)

意見の展開 (シンプル、説明可能):

- それぞれの意見には次のようなものがあります。
  - ステートメント
  - 自信 `c ∈ [0,1]`
  - last_updated
  - 証拠リンク (裏付けとなる事実と矛盾する事実 ID)
- 新しい事実が到着したとき:
  - エンティティの重複 + 類似性によって候補となる意見を見つける (最初に FTS、後で埋め込み)
  - 小さなデルタによって信頼度を更新します。大きなジャンプには強い矛盾と繰り返しの証拠が必要です

## CLI 統合: スタンドアロンと深い統合

推奨事項: **OpenClaw との緊密な統合** ただし、分離可能なコア ライブラリは保持します。

### なぜ OpenClaw に統合するのでしょうか?

- OpenClaw はすでに次のことを知っています。
  - ワークスペースのパス (`agents.defaults.workspace`)
  - セッションモデル + ハートビート
  - ロギング + トラブルシューティング パターン
- エージェント自体がツールを呼び出すようにしたい場合:
  - `openclaw memory recall "…" --k 25 --since 30d`
  - `openclaw memory reflect --since 7d`

### なぜライブラリを分割するのでしょうか?

- ゲートウェイ/ランタイムなしでメモリロジックをテスト可能に保つ
- 他のコンテキスト (ローカル スクリプト、将来のデスクトップ アプリなど) からの再利用

形状:
メモリ ツールは小規模な CLI + ライブラリ層を目的としていますが、これは探索的なものにすぎません。

## 「S-Collide」 / SuCo: いつ使用するか (研究)「S-Collide」が **SuCo (Subspace Collision)** を指す場合: これは、部分空間で学習/構造化された衝突を使用することにより、強力なリコール/レイテンシーのトレードオフをターゲットとする ANN 検索アプローチです (論文: arXiv 2411.14754、2024)

`~/.openclaw/workspace` の実際的な見解:

- **SuCo から始めないでください**。
- SQLite FTS + (オプション) 単純な埋め込みから始めます。 UX のほとんどはすぐに得られます。
- SuCo/HNSW/ScaNN クラスのソリューションを 1 回だけ検討します。
  - コーパスが大きい (数万/数十万のチャンク)
  - ブルートフォース埋め込み検索が遅すぎる
  - 語彙検索によって再現品質が大幅にボトルネックになっている

オフラインに適した代替手段 (複雑さが増しています):

- SQLite FTS5 + メタデータ フィルター (ゼロ ML)
- 埋め込み + ブルート フォース (チャンク数が少ない場合は驚くほどうまくいきます)
- HNSW インデックス (共通、堅牢、ライブラリ バインディングが必要)
- SuCo (研究グレード。埋め込み可能な確実な実装があれば魅力的)

  - Ollama をすでにお持ちの場合: ローカル モデルを埋め込みます。それ以外の場合は、ツールチェーンに小さな埋め込みモデルを同梱します。

## 最小の有用なパイロット

最小限のまだ役立つバージョンが必要な場合は、次のようにします。- 日次ログに `bank/` エンティティ ページと `## Retain` セクションを追加します。

- 引用 (パス + 行番号) を含むリコールには SQLite FTS を使用します。
- リコールの品質または規模が必要な場合にのみエンベディングを追加します。
