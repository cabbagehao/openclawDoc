# コマンドキュー (2026-01-16)

受信自動応答の実行 (すべてのチャネル) を小さなインプロセス キューを通じてシリアル化し、複数のエージェントの実行が衝突するのを防ぎながら、セッション間での安全な並列処理を可能にします。

## なぜ

- 自動応答の実行は高価になる可能性があり (LLM 呼び出し)、複数の受信メッセージが近くに到着すると衝突する可能性があります。
- シリアル化により、共有リソース (セッション ファイル、ログ、CLI 標準入力) の競合が回避され、アップストリームのレート制限が発生する可能性が減ります。

## 仕組み

- レーン対応 FIFO キューは、構成可能な同時実行数の上限を使用して各レーンを排出します (未構成のレーンのデフォルトは 1、メインのデフォルトは 4、サブエージェントは 8)。
- `runEmbeddedPiAgent` は **セッション キー** (レーン `session:<key>`) によってキューに登録され、セッションごとに 1 つのアクティブな実行のみが保証されます。
- 各セッションの実行は **グローバル レーン** (デフォルトでは `main`) のキューに入れられるため、全体の並列処理は `agents.defaults.maxConcurrent` によって制限されます。
- 詳細ログが有効になっている場合、キューに入れられた実行が開始前に 2 秒以上待機した場合、短い通知が発行されます。
- タイピング インジケーターはエンキュー時にすぐに起動するため (チャネルでサポートされている場合)、順番を待っている間もユーザー エクスペリエンスは変わりません。

## キューモード (チャネルごと)

受信メッセージは、現在の実行を制御したり、フォローアップの順番を待ったり、またはその両方を行うことができます。- `steer`: 現在の実行に直ちに挿入します (次のツール境界の後で保留中のツール呼び出しをキャンセルします)。ストリーミングしていない場合は、フォローアップに戻ります。

- `followup`: 現在の実行が終了した後、次のエージェント ターンのキューに入れます。
- `collect`: キューに入れられたすべてのメッセージを **単一**のフォローアップ ターンに結合します (デフォルト)。メッセージが異なるチャネル/スレッドをターゲットにしている場合、ルーティングを維持するためにメッセージは個別に排出されます。
- `steer-backlog` (別名 `steer+backlog`): 今すぐ操縦し、\*\*後続のターンのためにメッセージを保存します。
- `interrupt` (レガシー): そのセッションのアクティブな実行を中止し、最新のメッセージを実行します。
- `queue` (レガシー エイリアス): `steer` と同じ。

ステア バックログとは、ステアされた実行後にフォローアップの応答を得ることができることを意味します。
ストリーミング サーフェスは重複のように見える場合があります。必要に応じて `collect`/`steer` を優先してください
受信メッセージごとに 1 つの応答。
`/queue collect` をスタンドアロン コマンド (セッションごと) として送信するか、`messages.queue.byChannel.discord: "collect"` を設定します。

デフォルト (構成で未設定の場合):

- すべての表面 → `collect`

`messages.queue` を介してグローバルまたはチャネルごとに構成します。

```json5
{
  messages: {
    queue: {
      mode: "collect",
      debounceMs: 1000,
      cap: 20,
      drop: "summarize",
      byChannel: { discord: "collect" },
    },
  },
}
```

## キューのオプション

オプションは `followup`、`collect`、および `steer-backlog` (およびフォローアップにフォールバックする場合は `steer`) に適用されます。- `debounceMs`: フォローアップ ターンを開始する前に静かになるまで待ちます (「続行、続行」を防ぎます)。

- `cap`: セッションごとにキューに入れられるメッセージの最大数。
- `drop`: オーバーフロー ポリシー (`old`、`new`、`summarize`)。

Summarize は、ドロップされたメッセージの短い箇条書きリストを保持し、それを合成フォローアップ プロンプトとして挿入します。
デフォルト: `debounceMs: 1000`、`cap: 20`、`drop: summarize`。

## セッションごとのオーバーライド

- `/queue <mode>` をスタンドアロン コマンドとして送信し、現在のセッションのモードを保存します。
- オプションを組み合わせることができます: `/queue collect debounce:2s cap:25 drop:summarize`
- `/queue default` または `/queue reset` はセッション オーバーライドをクリアします。

## 範囲と保証

- ゲートウェイ応答パイプラインを使用するすべての受信チャネル (WhatsApp Web、Telegram、Slack、Discord、Signal、iMessage、Webchat など) で実行される自動応答エージェントに適用されます。
- デフォルト レーン (`main`) は、受信 + メイン ハートビートのプロセス全体です。 `agents.defaults.maxConcurrent` を設定して、複数のセッションを並行して実行できるようにします。
- 追加のレーンが存在する場合があります (例: `cron`、`subagent`)。そのため、受信応答をブロックすることなくバックグラウンド ジョブを並行して実行できます。
- セッションごとのレーンでは、一度に 1 つのエージェント実行のみが特定のセッションに接続されることが保証されます。
- 外部依存関係やバックグラウンド ワーカー スレッドはありません。純粋な TypeScript + Promise。

## トラブルシューティング- コマンドがスタックしているように見える場合は、詳細ログを有効にし、「queued for …ms」行を探してキューが空になっていることを確認します

- キューの深さが必要な場合は、詳細ログを有効にして、キューのタイミング ラインを監視します。
