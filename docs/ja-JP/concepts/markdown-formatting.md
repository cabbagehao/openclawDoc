# マークダウン形式

OpenClaw は、アウトバウンド Markdown を共有中間ファイルに変換してフォーマットします。
チャネル固有の出力をレンダリングする前の表現 (IR)。 IR は、
スタイル/リンク スパンを保持しながらソース テキストをそのまま維持できるため、チャンク化とレンダリングが可能になります。
チャネル間で一貫性を保ちます。

## 目標

- **一貫性:** 1 つの解析ステップ、複数のレンダラー。
- **安全なチャンク:** レンダリング前にテキストを分割するため、インライン書式設定は行われません
  チャンクをまたいで分割します。
- **チャネルフィット:** 同じ IR を Slack mrkdwn、Telegram HTML、および Signal にマッピングします。
  Markdown を再解析せずにスタイル範囲を変更します。

## パイプライン

1. **マークダウンの解析 -> IR**
   - IR は、プレーン テキストとスタイル スパン (太字/斜体/ストライク/コード/スポイラー) およびリンク スパンです。
   - オフセットは UTF-16 コード単位であるため、信号スタイルの範囲は API と一致します。
   - テーブルは、チャネルがテーブル変換を選択した場合にのみ解析されます。
2. **チャンク IR (フォーマットファースト)**
   - レンダリング前に IR テキストでチャンクが発生します。
   - インライン書式設定はチャンクに分割されません。スパンはチャンクごとにスライスされます。
3. **チャンネルごとのレンダリング**
   - **Slack:** mrkdwn トークン (太字/斜体/ストライク/コード)、`<url|label>` としてリンク。
   - **テレグラム:** HTML タグ (`<b>`、`<i>`、`<s>`、`<code>`、`<pre><code>`、`<a href>`)。
   - **信号:** プレーンテキスト + `text-style` 範囲。ラベルが異なる場合、リンクは `label (url)` になります。

## IR の例入力マークダウン

```markdown
Hello **world** — see [docs](https://docs.openclaw.ai).
```

IR (概略図):

```json
{
  "text": "Hello world — see docs.",
  "styles": [{ "start": 6, "end": 11, "style": "bold" }],
  "links": [{ "start": 19, "end": 23, "href": "https://docs.openclaw.ai" }]
}
```

## 使用される場所

- Slack、Telegram、および Signal アウトバウンド アダプターは IR からレンダリングします。
- 他のチャネル (WhatsApp、iMessage、MS Teams、Discord) では依然としてプレーン テキストまたは
  独自の書式設定ルール。Markdown テーブル変換が前に適用されます。
  チャンク化が有効になっている場合。

## テーブルの処理

マークダウン テーブルは、チャット クライアント間で一貫してサポートされていません。使用する
`markdown.tables` は、チャネルごと (およびアカウントごと) にコンバージョンを制御します。

- `code`: テーブルをコード ブロックとしてレンダリングします (ほとんどのチャネルのデフォルト)。
- `bullets`: 各行を箇条書きに変換します (Signal + WhatsApp のデフォルト)。
- `off`: テーブルの解析と変換を無効にします。生のテーブルテキストが通過します。

設定キー:

```yaml
channels:
  discord:
    markdown:
      tables: code
    accounts:
      work:
        markdown:
          tables: off
```

## チャンクルール

- チャンク制限はチャネル アダプター/構成から取得され、IR テキストに適用されます。
- コード フェンスは、末尾に改行が付いている単一のブロックとして保存されるため、チャネルは
  それらを正しくレンダリングします。
- リスト接頭辞とブロック引用接頭辞は IR テキストの一部であるため、チャンク化
  中間プレフィックスは分割しません。
- インライン スタイル (太字/斜体/ストライク/インラインコード/スポイラー) は分割されません。
  チャンク。レンダラーは各チャンク内のスタイルを再度開きます。

チャネル間のチャンク動作の詳細が必要な場合は、を参照してください。
[ストリーミング + チャンキング](/concepts/streaming)。

## リンクポリシー- **Slack:** `[label](url)` -> `<url|label>`;裸の URL は裸のままです。オートリンク

二重リンクを避けるために、解析中は無効になります。

- **テレグラム:** `[label](url)` -> `<a href="url">label</a>` (HTML 解析モード)。
- **シグナル:** `[label](url)` -> `label (url)` (ラベルが URL と一致しない場合)。

## ネタバレ

スポイラー マーカー (`||spoiler||`) は、Signal に対してのみ解析され、マッピングされます。
ネタバレスタイルの範囲。他のチャネルはそれらをプレーンテキストとして扱います。

## チャネル フォーマッタを追加または更新する方法

1. **一度解析します:** チャネルに適した共有 `markdownToIR(...)` ヘルパーを使用します。
   オプション (自動リンク、見出しスタイル、ブロック引用プレフィックス)。
2. **レンダリング:** `renderMarkdownWithMarkers(...)` と
   スタイル マーカー マップ (または信号スタイル範囲)。
3. **チャンク:** レンダリング前に `chunkMarkdownIR(...)` を呼び出します。各チャンクをレンダリングします。
4. **ワイヤー アダプター:** 新しいチャンカーを使用するようにチャネル送信アダプターを更新します。
   そしてレンダラー。
5. **テスト:** 形式テストと送信配信テストを追加または更新します。
   チャネルはチャンキングを使用します。

## よくある落とし穴- Slack 山括弧トークン (`<@U123>`、`<#C123>`、`<https://...>`) は次のとおりです

保存される。生の HTML を安全にエスケープします。

- Telegram HTML では、マークアップの破損を避けるために、タグの外側のテキストをエスケープする必要があります。
- 信号スタイルの範囲は UTF-16 オフセットに依存します。コードポイントオフセットを使用しないでください。
- 閉じマーカーが着地するように、フェンスで囲まれたコード ブロックの末尾の改行を保持します。
  自分たちのライン。
