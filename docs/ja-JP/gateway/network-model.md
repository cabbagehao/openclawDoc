ほとんどの操作は、単一の長時間実行されるゲートウェイ (`openclaw gateway`) を介して流れます。
チャネル接続と WebSocket コントロール プレーンを所有するプロセス。

## コアルール

- ホストごとに 1 つのゲートウェイが推奨されます。これは、WhatsApp Web セッションを所有できる唯一のプロセスです。レスキュー ボットまたは厳密な分離の場合は、分離されたプロファイルとポートを使用して複数のゲートウェイを実行します。 [複数のゲートウェイ](/gateway/multiple-gateways) を参照してください。
- ループバックを最初に行う: ゲートウェイ WS のデフォルトは `ws://127.0.0.1:18789` です。ウィザードは、ループバックの場合でも、デフォルトでゲートウェイ トークンを生成します。非ループバック バインドにはトークンが必要であるため、テールネット アクセスの場合は、`openclaw gateway --bind tailnet --token ...` を実行します。
- ノードは、必要に応じて LAN、テールネット、または SSH 経由でゲートウェイ WS に接続します。レガシー TCP ブリッジは非推奨になりました。
- キャンバス ホストは、ゲートウェイと **同じポート**上のゲートウェイ HTTP サーバーによって提供されます (デフォルト `18789`)。
  - `/__openclaw__/canvas/`
  - `/__openclaw__/a2ui/`
    `gateway.auth` が構成され、ゲートウェイがループバックを超えてバインドされる場合、これらのルートはゲートウェイ認証によって保護されます。ノード クライアントは、アクティブな WS セッションに関連付けられたノード スコープの機能 URL を使用します。 [ゲートウェイ構成](/gateway/configuration) (`canvasHost`、`gateway`) を参照してください。
- リモート使用は通常、SSH トンネルまたはテールネット VPN です。 [リモート アクセス](/gateway/remote) および [検出](/gateway/discovery) を参照してください。
