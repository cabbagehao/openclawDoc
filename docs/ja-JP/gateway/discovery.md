# 発見と輸送

OpenClaw には、表面的には似ているように見える 2 つの異なる問題があります。

1. **オペレーター リモート コントロール**: 他の場所で実行されているゲートウェイを制御する macOS メニュー バー アプリ。
2. **ノード ペアリング**: iOS/Android (および将来のノード) がゲートウェイを見つけて安全にペアリングします。

設計目標は、すべてのネットワーク検出/アドバタイズを **ノード ゲートウェイ** (`openclaw gateway`) に維持し、クライアント (Mac アプリ、iOS) をコンシューマーとして維持することです。

## 規約

- **ゲートウェイ**: 状態 (セッション、ペアリング、ノード レジストリ) を所有し、チャネルを実行する単一の長時間実行ゲートウェイ プロセス。ほとんどのセットアップでは、ホストごとに 1 つを使用します。分離されたマルチゲートウェイのセットアップが可能です。
- **ゲートウェイ WS (コントロール プレーン)**: デフォルトでは `127.0.0.1:18789` 上の WebSocket エンドポイント。 `gateway.bind` 経由で LAN/テールネットにバインドできます。
- **ダイレクト WS トランスポート**: LAN/テールネットに面したゲートウェイ WS エンドポイント (SSH なし)。
- **SSH トランスポート (フォールバック)**: SSH 経由で `127.0.0.1:18789` を転送することによるリモート制御。
- **レガシー TCP ブリッジ (非推奨/削除)**: 古いノード トランスポート ([ブリッジ プロトコル](/gateway/bridge-protocol) を参照)。発見のために宣伝されなくなりました。

プロトコルの詳細:

- [ゲートウェイプロトコル](/gateway/protocol)
- [ブリッジプロトコル(レガシー)](/gateway/bridge-protocol)

## 「直接」と SSH の両方を維持する理由- **Direct WS** は、同じネットワーク上およびテールネット内で最高の UX です

- Bonjour による LAN 上の自動検出
- ゲートウェイが所有するトークン + ACL のペアリング
- シェルへのアクセスは必要ありません。プロトコル表面をしっかりと監査可能に保つことができる
- **SSH** は引き続きユニバーサル フォールバックです。
  - SSH アクセスができる場所ならどこでも (無関係なネットワーク上でも) 動作します。
  - マルチキャスト/mDNS の問題を解決します。
  - SSH 以外に新しい受信ポートは必要ありません

## 検出入力 (クライアントがゲートウェイの場所を学習する方法)

### 1) Bonjour / mDNS (LAN のみ)

Bonjour はベストエフォート型であり、ネットワークを越えません。これは「同一 LAN」の便宜のためにのみ使用されます。

目標方向:

- **ゲートウェイ**は、Bonjour 経由で WS エンドポイントをアドバタイズします。
- クライアントは「ゲートウェイの選択」リストを参照して表示し、選択したエンドポイントを保存します。

トラブルシューティングとビーコンの詳細: [Bonjour](/gateway/bonjour)。

#### サービスビーコンの詳細- サービスの種類

- `_openclaw-gw._tcp` (ゲートウェイ トランスポート ビーコン)
- TXT キー (非秘密):
  - `role=gateway`
  - `lanHost=<hostname>.local`
  - `sshPort=22` (または宣伝されているもの)
  - `gatewayPort=18789` (ゲートウェイ WS + HTTP)
  - `gatewayTls=1` (TLS が有効な場合のみ)
  - `gatewayTlsSha256=<sha256>` (TLS が有効で、フィンガープリントが使用可能な場合のみ)
  - `canvasPort=<port>` (キャンバス ホスト ポート。現在、キャンバス ホストが有効な場合は `gatewayPort` と同じ)
  - `cliPath=<path>` (オプション; 実行可能な `openclaw` エントリポイントまたはバイナリへの絶対パス)
  - `tailnetDns=<magicdns>` (オプションのヒント; Tailscale が使用可能な場合に自動検出)

セキュリティに関する注意事項:

- Bonjour/mDNS TXT レコードは **未認証**です。クライアントは TXT 値を UX ヒントとしてのみ扱う必要があります。
- ルーティング (ホスト/ポート) では、TXT が提供する `lanHost`、`tailnetDns`、または `gatewayPort` よりも、**解決されたサービス エンドポイント** (SRV + A/AAAA) を優先する必要があります。
- TLS ピンニングでは、アドバタイズされた `gatewayTlsSha256` が以前に保存されたピンをオーバーライドすることを決して許可してはなりません。
- iOS/Android ノードは、検出ベースの直接接続を **TLS のみ**として扱い、初回 PIN (帯域外検証) を保存する前に明示的な「このフィンガープリントを信頼する」確認を要求する必要があります。

無効化/上書き:- `OPENCLAW_DISABLE_BONJOUR=1` は広告を無効にします。

- `~/.openclaw/openclaw.json` の `gateway.bind` は、ゲートウェイ バインド モードを制御します。
- `OPENCLAW_SSH_PORT` は、TXT でアドバタイズされる SSH ポート (デフォルトは 22) をオーバーライドします。
- `OPENCLAW_TAILNET_DNS` は、`tailnetDns` ヒント (MagicDNS) を公開します。
- `OPENCLAW_CLI_PATH` は、アドバタイズされた CLI パスをオーバーライドします。

### 2) テールネット (クロスネットワーク)

ロンドン/ウィーンスタイルのセットアップでは、Bonjour は役に立ちません。推奨される「直接」ターゲットは次のとおりです。

- Tailscale MagicDNS 名 (推奨) または安定したテールネット IP。

ゲートウェイが Tailscale で実行されていることを検出できる場合、クライアント (広域ビーコンを含む) に対するオプションのヒントとして `tailnetDns` を公開します。

### 3) 手動 / SSH ターゲット

ダイレクト ルートがない (またはダイレクトが無効になっている) 場合、クライアントはループバック ゲートウェイ ポートを転送することにより、いつでも SSH 経由で接続できます。

[リモート アクセス](/gateway/remote) を参照してください。

## トランスポートの選択 (クライアント ポリシー)

推奨されるクライアントの動作:

1. ペアになっている直接エンドポイントが構成されており、到達可能な場合は、それを使用します。
2. それ以外の場合、Bonjour が LAN 上でゲートウェイを見つけた場合は、ワンタップで「このゲートウェイを使用する」を選択し、それを直接エンドポイントとして保存します。
3. それ以外の場合、テールネット DNS/IP が設定されている場合は、直接試してください。
4. それ以外の場合は、SSH にフォールバックします。

## ペアリング + 認証 (ダイレクトトランスポート)

ゲートウェイは、ノード/クライアントの許可に関する信頼できる情報源です。- ペアリング要求はゲートウェイで作成/承認/拒否されます ([ゲートウェイ ペアリング](/gateway/pairing) を参照)。

- ゲートウェイは以下を強制します。
  - 認証 (トークン/キーペア)
  - スコープ/ACL (ゲートウェイはすべてのメソッドに対する生のプロキシではありません)
  - レート制限

## コンポーネントごとの責任

- **ゲートウェイ**: 検出ビーコンをアドバタイズし、ペアリング決定を所有し、WS エンドポイントをホストします。
- **macOS アプリ**: ゲートウェイの選択を支援し、ペアリング プロンプトを表示し、SSH をフォールバックとしてのみ使用します。
- **iOS/Android ノード**: 便利なように Bonjour を参照し、ペアリングされた Gateway WS に接続します。
