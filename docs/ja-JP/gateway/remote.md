# リモート アクセス (SSH、トンネル、テールネット)

このリポジトリは、単一のゲートウェイ (マスター) を専用ホスト (デスクトップ/サーバー) 上で実行し、それにクライアントを接続することにより、「SSH 経由のリモート」をサポートします。

- **オペレーター (あなた/macOS アプリ)** の場合: SSH トンネリングはユニバーサル フォールバックです。
- **ノード (iOS/Android および将来のデバイス)** の場合: ゲートウェイ **WebSocket** (必要に応じて LAN/テールネットまたは SSH トンネル) に接続します。

## 核となるアイデア

- ゲートウェイ WebSocket は、構成されたポート上の **ループバック** にバインドします (デフォルトは 18789)。
- リモートで使用する場合は、ループバック ポートを SSH 経由で転送します (またはテールネット/VPN を使用し、トンネルを使用しません)。

## 一般的な VPN/テールネット設定 (エージェントが存在する場所)

**ゲートウェイ ホスト** を「エージェントが存在する場所」と考えてください。セッション、認証プロファイル、チャネル、および状態を所有します。
ラップトップ/デスクトップ (およびノー​​ド) がそのホストに接続します。

### 1) テールネットの常時接続ゲートウェイ (VPS またはホーム サーバー)

永続ホスト上でゲートウェイを実行し、**Tailscale** または SSH 経由でアクセスします。

- **最高の UX:** `gateway.bind: "loopback"` を維持し、コントロール UI に **Tailscale Serve** を使用します。
- **フォールバック:** アクセスが必要なマシンからのループバック + SSH トンネルを維持します。
- **例:** [exe.dev](/install/exe-dev) (簡易 VM) または [Hetzner](/install/hetzner) (実稼働 VPS)。

これは、ラップトップが頻繁にスリープ状態になるが、エージェントを常にオンにしたい場合に最適です。

### 2) 自宅のデスクトップはゲートウェイを実行し、ラップトップはリモート制御しますラップトップはエージェントを**実行しません**。リモートで接続します

- macOS アプリの **SSH 経由のリモート** モードを使用します ([設定] → [一般] → [OpenClaw を実行する])。
- アプリがトンネルを開いて管理するため、WebChat + ヘルスチェックが「正常に機能」します。

ランブック: [macOS リモート アクセス](/platforms/mac/remote)。

### 3) ラップトップがゲートウェイを実行し、他のマシンからリモート アクセスします

ゲートウェイをローカルに保ちますが、安全に公開します。

- 他のマシンからラップトップへの SSH トンネル、または
- Tailscale コントロール UI を提供し、ゲートウェイ ループバックのみを維持します。

ガイド: [Tailscale](/gateway/tailscale) および [Web 概要](/web)。

## コマンド フロー (何がどこで実行されるか)

1 つのゲートウェイ サービスが状態とチャネルを所有します。ノードはペリフェラルです。

フロー例（テレグラム→ノード）：

- 電報メッセージが **ゲートウェイ** に到着します。
- ゲートウェイは **エージェント** を実行し、ノード ツールを呼び出すかどうかを決定します。
- ゲートウェイは、ゲートウェイ WebSocket (`node.*` RPC) 経由で **ノード** を呼び出します。
- ノードは結果を返します。ゲートウェイは Telegram に応答します。

注:

- **ノードはゲートウェイ サービスを実行しません。** 意図的に分離プロファイルを実行しない限り、ホストごとに 1 つのゲートウェイのみを実行する必要があります ([複数のゲートウェイ](/gateway/multiple-gateways) を参照)。
- macOS アプリの「ノード モード」は、ゲートウェイ WebSocket 上の単なるノード クライアントです。

## SSH トンネル (CLI + ツール)

リモート ゲートウェイ WS へのローカル トンネルを作成します。

```bash
ssh -N -L 18789:127.0.0.1:18789 user@host
```

トンネルが開いている状態:- `openclaw health` および `openclaw status --deep` は、`ws://127.0.0.1:18789` 経由でリモート ゲートウェイに到達するようになりました。

- `openclaw gateway {status,health,send,agent,call}` は、必要に応じて `--url` 経由で転送された URL をターゲットにすることもできます。

注: `18789` を、構成済みの `gateway.port` (または `--port`/`OPENCLAW_GATEWAY_PORT`) に置き換えます。
注: `--url` を渡すと、CLI は構成または環境の資格情報にフォールバックしません。
`--token` または `--password` を明示的に含めます。明示的な資格情報が欠落しているとエラーになります。

## CLI リモートのデフォルト

リモート ターゲットを永続化して、CLI コマンドがデフォルトでそれを使用するようにすることができます。

```json5
{
  gateway: {
    mode: "remote",
    remote: {
      url: "ws://127.0.0.1:18789",
      token: "your-token",
    },
  },
}
```

ゲートウェイがループバック専用の場合は、URL を `ws://127.0.0.1:18789` のままにして、最初に SSH トンネルを開きます。

## 資格情報の優先順位

ゲートウェイの認証情報の解決は、コール/プローブ/ステータス パス、Discord の実行承認モニタリング、およびノードとホストの接続にわたる 1 つの共有コントラクトに従います。- 明示的な認証情報 (`--token`、`--password`、またはツール `gatewayToken`) は、明示的な認証を受け入れる呼び出しパスでは常に優先されます。

- URL オーバーライドの安全性:
  - CLI URL オーバーライド (`--url`) は、暗黙的な config/env 認証情報を再利用しません。
  - 環境 URL オーバーライド (`OPENCLAW_GATEWAY_URL`) では、環境資格情報のみを使用できます (`OPENCLAW_GATEWAY_TOKEN` / `OPENCLAW_GATEWAY_PASSWORD`)。
- ローカル モードのデフォルト:
  - トークン: `OPENCLAW_GATEWAY_TOKEN` -> `gateway.auth.token` -> `gateway.remote.token`
  - パスワード: `OPENCLAW_GATEWAY_PASSWORD` -> `gateway.auth.password` -> `gateway.remote.password`
- リモート モードのデフォルト:
  - トークン: `gateway.remote.token` -> `OPENCLAW_GATEWAY_TOKEN` -> `gateway.auth.token`
  - パスワード: `OPENCLAW_GATEWAY_PASSWORD` -> `gateway.remote.password` -> `gateway.auth.password`
- リモート プローブ/ステータス トークン チェックはデフォルトで厳密です。リモート モードをターゲットとする場合、`gateway.remote.token` のみを使用します (ローカル トークン フォールバックはありません)。
- レガシー `CLAWDBOT_GATEWAY_*` 環境変数は、互換性コール パスでのみ使用されます。プローブ/ステータス/認証の解決には `OPENCLAW_GATEWAY_*` のみが使用されます。

## SSH 経由のチャット UI

WebChat は別の HTTP ポートを使用しなくなりました。 SwiftUI チャット UI は、ゲートウェイ WebSocket に直接接続します。

- SSH 経由で `18789` を転送し (上記を参照)、クライアントを `ws://127.0.0.1:18789` に接続します。
- macOS では、トンネルを自動的に管理するアプリの「Remote over SSH」モードを優先します。

## macOS アプリ「SSH 経由のリモート」macOS メニュー バー アプリは、同じセットアップをエンドツーエンドで実行できます (リモート ステータス チェック、Web チャット、および音声ウェイク転送)

ランブック: [macOS リモート アクセス](/platforms/mac/remote)。

## セキュリティ ルール (リモート/VPN)

短いバージョン: バインドが必要であることが確実でない限り、**ゲートウェイ ループバックのみ**を維持してください。

- **ループバック + SSH/Tailscale Serve** が最も安全なデフォルトです (公開なし)。
- プレーンテキスト `ws://` は、デフォルトではループバックのみです。信頼できるプライベート ネットワークの場合、
  クライアント プロセスに `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` をブレークグラスとして設定します。
- **非ループバック バインド** (`lan`/`tailnet`/`custom`、またはループバックが使用できない場合は `auto`) は認証トークン/パスワードを使用する必要があります。
- `gateway.remote.token` / `.password` はクライアント資格情報ソースです。サーバー認証を独自に構成することは**ありません**。
- ローカル呼び出しパスは、`gateway.auth.*` が設定されていない場合、フォールバックとして `gateway.remote.*` を使用できます。
- `gateway.remote.tlsFingerprint` は、`wss://` を使用するときにリモート TLS 証明書を固定します。
- **Tailscale Serve** は ID 経由でコントロール UI/WebSocket トラフィックを認証できます
  `gateway.auth.allowTailscale: true` の場合のヘッダー。 HTTP API エンドポイントは引き続き
  トークン/パスワード認証が必要です。このトークンレス フローは、ゲートウェイ ホストが
  信頼されています。どこでもトークン/パスワードが必要な場合は、`false` に設定します。
- ブラウザ制御をオペレータ アクセスのように扱います: テールネットのみ + 意図的なノード ペア。

詳細: [セキュリティ](/gateway/security)。
