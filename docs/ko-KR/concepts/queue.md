# 명령 큐 (2026-01-16)

여러 세션 사이에서는 안전한 병렬성을 허용하면서도 여러 에이전트 실행이 서로 충돌하지 않도록, 아주 작은 인프로세스 큐를 통해 들어오는 자동 응답 실행(모든 채널)을 직렬화합니다.

## 이유

- 자동 응답 실행은 비용이 클 수 있으며(LLM 호출), 들어오는 메시지 여러 개가 거의 동시에 도착하면 서로 충돌할 수 있습니다.
- 직렬화하면 공유 리소스(세션 파일, 로그, CLI stdin)를 두고 경쟁하는 일을 피할 수 있고 업스트림 rate limit에 걸릴 가능성도 줄어듭니다.

## 동작 방식

- 레인을 인식하는 FIFO 큐가 각 레인을 설정 가능한 동시성 상한으로 소진합니다(설정되지 않은 레인의 기본값은 1, `main`의 기본값은 4, `subagent`의 기본값은 8).
- `runEmbeddedPiAgent`는 **세션 키**(레인 `session:<key>`) 기준으로 큐에 넣어 세션당 활성 실행이 하나만 있도록 보장합니다.
- 그런 다음 각 세션 실행은 전체 병렬성이 `agents.defaults.maxConcurrent`로 제한되도록 **전역 레인**(기본값은 `main`)에 큐잉됩니다.
- 자세한 로그를 켜면, 시작하기 전에 약 2초를 넘게 기다린 큐잉된 실행은 짧은 안내를 출력합니다.
- 타이핑 인디케이터는 여전히 큐에 넣는 즉시(채널이 지원하는 경우) 발생하므로, 순서를 기다리는 동안에도 사용자 경험은 바뀌지 않습니다.

## 큐 모드(채널별)

들어오는 메시지는 현재 실행을 조종하거나, 후속 턴을 기다리거나, 둘 다 할 수 있습니다.

- `steer`: 현재 실행에 즉시 주입합니다(다음 도구 경계 뒤에 대기 중인 도구 호출을 취소함). 스트리밍 중이 아니면 `followup`으로 폴백합니다.
- `followup`: 현재 실행이 끝난 뒤 다음 에이전트 턴에 들어가도록 큐에 넣습니다.
- `collect`: 큐에 쌓인 모든 메시지를 **단일** 후속 턴으로 합칩니다(기본값). 메시지가 서로 다른 채널/스레드를 대상으로 하면 라우팅을 보존하기 위해 각각 따로 소진됩니다.
- `steer-backlog`(별칭 `steer+backlog`): 지금 `steer`하고 **동시에** 후속 턴용 메시지도 보존합니다.
- `interrupt`(레거시): 해당 세션의 활성 실행을 중단한 뒤 가장 최신 메시지를 실행합니다.
- `queue`(레거시 별칭): `steer`와 같습니다.

Steer-backlog는 조종된 실행 뒤에 후속 응답이 하나 더 올 수 있다는 뜻이므로,
스트리밍 UI에서는 중복처럼 보일 수 있습니다. 들어오는 메시지당 응답 하나를 원한다면 `collect` 또는 `steer`를 선호하세요.
독립 실행형 명령으로 `/queue collect`를 보내 세션별로 설정하거나, `messages.queue.byChannel.discord: "collect"`를 설정하세요.

기본값(설정에서 지정하지 않았을 때):

- 모든 표면 → `collect`

전역 또는 채널별로 `messages.queue`에서 설정합니다.

```json5
{
  messages: {
    queue: {
      mode: "collect",
      debounceMs: 1000,
      cap: 20,
      drop: "summarize",
      byChannel: { discord: "collect" },
    },
  },
}
```

## 큐 옵션

옵션은 `followup`, `collect`, `steer-backlog`에 적용되며(`steer`가 `followup`으로 폴백할 때도 적용됨):

- `debounceMs`: 후속 턴을 시작하기 전에 잠잠해질 때까지 기다립니다("continue, continue"를 방지함).
- `cap`: 세션당 큐에 담을 수 있는 최대 메시지 수입니다.
- `drop`: 초과 시 처리 정책입니다(`old`, `new`, `summarize`).

`summarize`는 드롭된 메시지의 짧은 불릿 목록을 유지하고 그것을 합성된 후속 프롬프트로 주입합니다.
기본값은 `debounceMs: 1000`, `cap: 20`, `drop: summarize`입니다.

## 세션별 재정의

- 독립 실행형 명령으로 `/queue <mode>`를 보내 현재 세션의 모드를 저장합니다.
- 옵션을 함께 조합할 수 있습니다: `/queue collect debounce:2s cap:25 drop:summarize`
- `/queue default` 또는 `/queue reset`은 세션 재정의를 지웁니다.

## 범위와 보장

- 게이트웨이 응답 파이프라인을 사용하는 모든 인바운드 채널(WhatsApp web, Telegram, Slack, Discord, Signal, iMessage, webchat 등)의 자동 응답 에이전트 실행에 적용됩니다.
- 기본 레인(`main`)은 인바운드와 메인 하트비트 전반에 걸쳐 프로세스 전체에서 공유됩니다. 여러 세션을 병렬로 허용하려면 `agents.defaults.maxConcurrent`를 설정하세요.
- 추가 레인(예: `cron`, `subagent`)이 존재할 수 있으므로 백그라운드 작업은 들어오는 응답을 막지 않고 병렬로 실행될 수 있습니다.
- 세션별 레인은 특정 세션을 동시에 둘 이상의 에이전트 실행이 건드리지 않도록 보장합니다.
- 외부 의존성이나 백그라운드 워커 스레드는 없으며, 순수 TypeScript + promise만 사용합니다.

## 문제 해결

- 명령이 멈춘 것처럼 보이면 자세한 로그를 켜고 `"queued for …ms"` 줄을 찾아 큐가 실제로 소진되고 있는지 확인하세요.
- 큐 깊이가 필요하면 자세한 로그를 켜고 큐 타이밍 줄을 확인하세요.
