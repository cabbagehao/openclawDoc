# 시스템 보안 (Security) 🔒

> \[!WARNING]
> **개인 비서 신뢰 모델:** 본 가이드는 Gateway당 신뢰할 수 있는 단일 운영자 경계(단일 사용자/개인 비서 모델)를 전제로 함.
> OpenClaw는 다수의 적대적 사용자가 단일 에이전트/Gateway를 공유하는 환경을 위한 **멀티테넌트(Multi-tenant) 보안 경계를 제공하지 않음.**
> 서로 신뢰할 수 없는 사용자 환경이 필요하다면 반드시 신뢰 경계를 분리해야 함 (별도 Gateway 실행 및 자격 증명 사용, 가급적 OS 사용자 및 호스트 분리 권장).

## 보안 범위 정의: 개인 비서 모델

OpenClaw의 보안 지침은 **개인 비서** 배포 환경을 가정함. 즉, 신뢰할 수 있는 운영자 경계는 하나이며, 그 아래에 여러 에이전트가 존재할 수 있는 구조임.

* **지원되는 보안 태세**: Gateway당 단일 사용자/신뢰 경계 유지 (경계마다 별도의 OS 사용자/호스트/VPS 사용 권장).
* **지원되지 않는 환경**: 서로 신뢰하지 않거나 적대적인 사용자가 하나의 Gateway/에이전트를 공유하는 구성.
* **다중 사용자 대응**: 신뢰할 수 없는 여러 사용자가 도구가 활성화된 에이전트 하나를 사용하는 경우, 해당 에이전트에 위임된 모든 도구 권한을 사용자들이 공유하는 것으로 간주해야 함.

본 문서는 이러한 **개인 비서 모델 내부에서의 보안 강화(Hardening)** 방법을 설명하며, 공유 Gateway 상에서의 완벽한 멀티테넌트 격리를 보장하지 않음.

## 빠른 보안 점검: `openclaw security audit`

참고: [보안 모델의 정식 검증 절차](/security/formal-verification/)

설정을 변경하거나 네트워크 노출 범위를 넓힌 후에는 반드시 정기적으로 감사를 실행하기 바람:

```bash
openclaw security audit
openclaw security audit --deep
openclaw security audit --fix
openclaw security audit --json
```

이 명령어는 Gateway 인증 노출, 브라우저 제어 취약점, 권한 상승 허용 목록 오류, 파일 시스템 권한 문제 등 흔히 발생하는 실수들을 자동으로 탐지함.

**완벽하게 안전한 구성이란 존재하지 않음.** 목표는 다음 사항들을 명확히 결정하고 관리하는 것임:

* 누가 봇과 대화할 수 있는가?
* 봇이 시스템의 어느 범위까지 제어할 수 있는가?
* 봇이 어떤 데이터에 접근할 수 있는가?

최소한의 접근 권한으로 시작하여, 신뢰가 확보된 경우에만 점진적으로 범위를 확장할 것을 권장함.

## 배포 시 전제 조건 (중요)

OpenClaw는 Gateway 호스트 및 설정 파일 경계가 신뢰된다고 가정함:

* Gateway 호스트 상태나 설정(`~/.openclaw/openclaw.json`)을 수정할 수 있는 사용자는 모두 **신뢰된 운영자**로 간주함.
* 서로 신뢰하지 않는 여러 운영자를 위해 단일 Gateway를 운영하는 방식은 **권장되지 않음.**
* 팀 단위 운영 시에는 신뢰 경계별로 Gateway를 분리해야 함 (최소한 OS 사용자나 호스트 분리).
* 권장 기본값: 호스트(또는 VPS)당 사용자 1명, 해당 사용자용 Gateway 1개, 그 내부의 에이전트 1개 이상 구성.

### 실질적 의미 (운영자 신뢰 경계)

단일 Gateway 인스턴스 내에서 인증된 운영자 액세스는 사용자별로 격리된 테넌트 역할이 아닌, **시스템 전체를 제어하는 신뢰된 역할**임.

* 읽기/제어 권한을 가진 운영자는 설계상 모든 세션 메타데이터와 대화 이력을 열람할 수 있음.
* `sessionKey`, 세션 ID 등은 메시지 라우팅을 위한 식별자일 뿐, 사용자 간의 보안 격리 토큰이 아님.
* 따라서 `sessions.list`, `chat.history` 등에서 운영자 간의 완벽한 격리를 기대하는 것은 본 보안 모델의 범위를 벗어남.

## 개인 비서 모델의 특성 (멀티테넌트 아님)

* 여러 사람이 하나의 도구 사용 에이전트와 대화할 수 있다면, 각 사용자 모두가 해당 에이전트의 권한 세트를 조종할 수 있음.
* 사용자별 세션/메모리 격리는 개인정보 보호에는 도움이 되지만, 공유 에이전트를 개별 사용자별 호스트 권한 경계로 변환시키지는 않음.

### 공유 Slack 워크스페이스의 위험성

"Slack의 모든 구성원이 봇과 대화할 수 있는" 환경에서의 핵심 위험은 **도구 권한의 무분별한 위임**임:

* 허용된 모든 발신자가 에이전트 정책 내에서 도구 호출(`exec`, 브라우저, 네트워크 등)을 유도할 수 있음.
* 특정 발신자의 프롬프트 주입 공격이 시스템 상태나 장치 제어에 영향을 줄 수 있음.
* 공유 에이전트가 민감한 자격 증명을 보유한 경우, 도구 사용을 통한 정보 유출 위험이 존재함.

팀 워크플로에는 최소한의 도구만 허용한 별도 에이전트를 할당하고, 개인용 에이전트는 비공개로 운영하기 바람.

## Gateway 및 노드 기기 신뢰 개념

Gateway와 노드 기기는 서로 다른 역할을 수행하는 하나의 신뢰 도메인으로 취급함:

* **Gateway**: 제어 플레인 및 정책 결정 지점 (`gateway.auth`, 도구 정책, 라우팅).
* **노드 기기**: Gateway에 페어링된 원격 실행 지점 (기기 명령 실행, 로컬 역량 제공).
* Gateway 인증을 통과한 호출자는 시스템 전체 스코프에서 신뢰됨. 페어링 완료 후의 노드 동작은 해당 노드에서의 신뢰된 운영자 동작으로 간주됨.
* 실행 승인 절차(Allowlist + Ask)는 운영자의 의도를 재확인하는 안전장치(Guardrail)이지, 악의적인 다중 사용자를 막기 위한 기술적 격리 수단이 아님.

## 신뢰 경계 매트릭스

| 제어 항목              | 의미                        | 오해하기 쉬운 부분                      |
| :----------------- | :------------------------ | :------------------------------ |
| **`gateway.auth`** | Gateway API 접속을 위한 호출자 인증 | "모든 메시지 프레임마다 개별 서명이 필요함"       |
| **`sessionKey`**   | 대화 맥락 및 세션 선택을 위한 라우팅 키   | "세션 키 자체가 사용자 인증 경계임"           |
| **프롬프트 안전장치**      | 모델 오용 및 악용 위험 완화          | "프롬프트 인젝션 성공이 곧 인증 우회를 의미함"     |
| **캔버스/브라우저 실행**    | 의도된 운영자용 실행 역량            | "어떠한 JS 실행 환경도 이 모델에서는 취약점임"    |
| **노드 페어링 및 명령**    | 페어링된 기기에서의 운영자급 원격 실행     | "원격 기기 제어는 기본적으로 비신뢰 접근으로 봐야 함" |

## 설계상 취약점으로 간주하지 않는 케이스

* 정책/인증/샌드박스 우회가 동반되지 않은 단순 프롬프트 인젝션 사례.
* 동일 호스트/설정을 공유하면서 악의적인 다중 사용자 운영을 가정한 주장.
* 공유 Gateway 환경에서 정상적인 운영자의 읽기 권한(세션 목록 등)을 권한 우회(IDOR)로 분류하는 사례.
* 루프백 전용 Gateway에서의 HSTS 부재 등 로컬 전용 배포 관련 지적.
* `sessionKey`를 사용자 인증 토큰으로 오해하여 제기된 "사용자별 권한 관리 부재" 지적.

## 60초 보안 기본선 (Hardened Baseline)

가장 안전한 초기 설정을 적용한 후, 필요한 도구만 에이전트별로 허용하기 바람:

```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    auth: { mode: "token", token: "강력하고-무작위한-토큰-생성" },
  },
  session: {
    dmScope: "per-channel-peer", // DM 세션 격리
  },
  tools: {
    profile: "messaging",
    deny: ["group:automation", "group:runtime", "group:fs", "sessions_spawn", "sessions_send"],
    fs: { workspaceOnly: true },
    exec: { security: "deny", ask: "always" },
    elevated: { enabled: false },
  },
  channels: {
    whatsapp: { dmPolicy: "pairing", groups: { "*": { requireMention: true } } },
  },
}
```

이 구성은 Gateway를 로컬로 격리하고, 개인 대화를 사용자별로 분리하며, 민감한 제어/실행 도구를 기본적으로 차단함.

## 공유 인박스 운영 규칙

둘 이상의 사용자가 봇에게 DM을 보낼 수 있는 경우:

* `session.dmScope`를 `"per-channel-peer"`로 설정하여 대화 맥락을 격리함.
* `dmPolicy`를 `"pairing"`으로 유지하거나 엄격한 허용 목록을 관리함.
* **주의**: 이는 협업을 위한 인박스 강화 조치일 뿐이며, 호스트 제어권을 가진 사용자 간의 적대적 격리를 목적으로 설계된 것은 아님.

## 자격 증명 저장 위치 가이드

보안 감사 및 백업 시 다음 경로를 참고함:

* **WhatsApp**: `~/.openclaw/credentials/whatsapp/<accountId>/creds.json`
* **채널 토큰**: 설정 파일, 환경 변수 또는 시크릿 참조(SecretRef) 사용.
* **페어링 허용 목록**: `~/.openclaw/credentials/<channel>-allowFrom.json`.
* **모델 인증 프로필**: `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`.
* **시크릿 페이로드**: `~/.openclaw/secrets.json` (선택 사항).

## 보안 감사 주요 항목 (Glossary)

| 체크 ID                         | 위험 수준  | 의미 및 영향                | 조치 방법                         |
| :---------------------------- | :----- | :--------------------- | :---------------------------- |
| `fs.state_dir.perms_writable` | **심각** | 타인이 OpenClaw 상태를 수정 가능 | 디렉터리 권한 `700`으로 조정            |
| `gateway.bind_no_auth`        | **심각** | 인증 없이 외부 네트워크 노출       | `gateway.auth.*` 설정 필수        |
| `gateway.tailscale_funnel`    | **심각** | 공용 인터넷에 즉시 노출됨         | 사용 목적 재확인 및 격리                |
| `logging.redact_off`          | 경고     | 로그에 민감한 정보가 평문 노출      | `logging.redactSensitive` 활성화 |
| `sandbox.docker_config_off`   | 경고     | 샌드박스 설정은 있으나 미사용 중     | `sandbox.mode` 활성화 검토         |

## 프롬프트 인젝션 (Prompt Injection) 대응

* **인바운드 제어**: DM 페어링 및 허용 목록을 엄격히 관리함.
* **멘션 게이팅**: 그룹 대화에서는 항상 봇을 멘션해야 응답하도록 설정함.
* **외부 콘텐츠 격리**: 봇이 읽는 웹 페이지, 이메일, 문서 등은 항상 '적대적'일 수 있다고 가정함.
* **샌드박싱 활용**: 도구 실행은 반드시 격리된 샌드박스에서 수행하고, 호스트의 시크릿 정보 접근을 차단함.
* **모델 선택**: 구형/소형 모델은 인젝션 공격에 매우 취약함. 도구 사용 에이전트에는 반드시 최신 고사양 모델을 사용함.

## 런타임 보안 강화

### 1) 파일 권한 관리

* `openclaw.json`: `600` (소유자만 읽기/쓰기 가능)
* `~/.openclaw`: `700` (소유자만 접근 가능)

### 2) 네트워크 격리

* **루프백 권장**: 가급적 `gateway.bind: "loopback"` 상태를 유지함.
* **Tailscale 활용**: 외부 접속이 필요한 경우 LAN 노출보다는 Tailscale Serve를 통한 보안 터널링을 권장함.
* **역방향 프록시**: 프록시(Nginx 등) 사용 시 `gateway.trustedProxies` 설정을 통해 클라이언트 IP 위조를 방지함.

### 3) 브라우저 제어 보안

* 에이전트 전용 브라우저 프로필 사용 (개인 프로필 연결 금지).
* 샌드박스 내 브라우저 구동 및 호스트 제어 기능 비활성화 권장.
* 브라우저를 통한 사설망 접근(SSRF)을 차단함 (`dangerouslyAllowPrivateNetwork: false`).

## 침해 사고 대응 (Incident Response)

사고 발생 시 즉시 다음 조치를 수행함:

1. **차단**: Gateway 프로세스를 즉시 종료하고 네트워크 노출을 차단함.
2. **회전**: Gateway 토큰, API 키, 채널 자격 증명 등 유출이 의심되는 모든 비밀 정보를 갱신함.
3. **감사**: 로그 및 세션 이력을 검토하여 피해 범위를 파악함. `openclaw security audit --deep` 실행.
