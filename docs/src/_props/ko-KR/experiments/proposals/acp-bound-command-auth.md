# ACP 바인딩 명령 권한 부여 (제안)

상태: 제안됨. **아직 구현되지 않았다.**

이 문서는 ACP 바인딩 대화에서 네이티브 명령에 적용할 장기 권한 모델을 설명한다. 실험용 제안 문서이며, 현재 운영 동작을 대체하지 않는다.

구현된 현재 동작은 다음 소스와 테스트를 참고한다.

* `src/telegram/bot-native-commands.ts`
* `src/discord/monitor/native-command.ts`
* `src/auto-reply/reply/commands-core.ts`

## 문제

오늘날 우리는 `/new`, `/reset` 같은 명령별 검사에 의존하고 있으며, allowlist 가 비어 있어도 ACP 바인딩 채널/토픽 내부에서 동작해야 한다. 이는 당장의 UX 문제를 해결하지만, 명령 이름 기반 예외는 확장성이 없다.

## 장기 방향

명령 권한 부여를 임시 핸들러 로직에서 명령 메타데이터 + 공통 정책 평가기로 옮긴다.

### 1) 명령 정의에 권한 정책 메타데이터 추가

각 명령 정의는 권한 정책을 선언해야 한다. 예시 형태:

```ts
type CommandAuthPolicy =
  | { mode: "owner_or_allowlist" } // 기본값, 현재의 엄격한 동작
  | { mode: "bound_acp_or_owner_or_allowlist" } // 명시적으로 바인딩된 ACP 대화에서 허용
  | { mode: "owner_only" };
```

`/new`, `/reset` 은 `bound_acp_or_owner_or_allowlist` 를 사용한다.
대부분의 다른 명령은 `owner_or_allowlist` 를 유지한다.

### 2) 채널 전반에서 하나의 평가기 공유

다음 요소를 사용해 명령 권한을 평가하는 헬퍼를 하나 도입한다.

* 명령 정책 메타데이터
* 발신자 권한 상태
* 해석된 대화 바인딩 상태

동작 드리프트를 막기 위해 Telegram 과 Discord 의 네이티브 핸들러는 동일한 헬퍼를 호출해야 한다.

### 3) 바인딩 일치를 우회 경계로 사용

정책이 바인딩된 ACP 우회를 허용하더라도, 현재 대화에 대해 구성된 바인딩 일치가 실제로 해석된 경우에만 권한을 부여한다. 단지 현재 세션 키가 ACP 처럼 보인다는 이유만으로는 허용하지 않는다.

이렇게 하면 경계가 명시적으로 유지되고, 의도치 않은 완화 확대를 최소화할 수 있다.

## 왜 더 나은가

* 앞으로 명령이 늘어나도 명령 이름 조건문을 추가하지 않아도 된다.
* 채널 간 동작을 일관되게 유지할 수 있다.
* 명시적 바인딩 일치를 요구하므로 현재 보안 모델을 유지한다.
* allowlist 를 보편적 의무가 아니라 선택적 강화 수단으로 남겨둘 수 있다.

## 롤아웃 계획(향후)

1. 명령 레지스트리 타입과 명령 데이터에 명령 권한 정책 필드를 추가한다.
2. 공통 평가기를 구현하고 Telegram + Discord 네이티브 핸들러를 마이그레이션한다.
3. `/new`, `/reset` 을 메타데이터 기반 정책으로 옮긴다.
4. 정책 모드와 채널 표면별 테스트를 추가한다.

## 비목표

* 이 제안은 ACP 세션 라이프사이클 동작을 바꾸지 않는다.
* 이 제안은 모든 ACP 바인딩 명령에 allowlist 를 요구하지 않는다.
* 이 제안은 기존 라우트 바인딩 의미론을 바꾸지 않는다.

## 메모

* 이 제안은 의도적으로 추가적인 문서이며, 기존 실험 문서를 삭제하거나 대체하지 않는다.
