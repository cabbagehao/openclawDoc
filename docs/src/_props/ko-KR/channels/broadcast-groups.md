# 브로드캐스트 그룹 (Broadcast Groups)

**상태**: 실험적 기능 (Experimental)\
**버전**: v2026.1.9에 추가됨.

## 개요

브로드캐스트 그룹은 여러 에이전트가 동일한 수신 메시지를 동시에 처리하고 응답할 수 있게 하는 기능임. 이를 통해 하나의 WhatsApp 그룹이나 DM 내에서 하나의 전화번호만 사용하여 특정 분야에 특화된 여러 에이전트 팀이 협업하는 환경을 구축할 수 있음.

현재 지원 범위: **WhatsApp 전용** (웹 채널).

브로드캐스트 그룹은 채널 허용 목록(Allowlist) 및 그룹 활성화 규칙이 먼저 평가된 후 적용됨. 즉, WhatsApp 그룹 설정상 에이전트가 응답해야 하는 상황(예: 멘션 발생 시)에 브로드캐스트가 실행됨.

## 주요 사용 사례

### 1. 특화된 에이전트 팀 운영

각기 다른 전문 분야를 가진 에이전트들을 한곳에 배치하여 협업하도록 함:

* **코드 리뷰어**: 코드 스니펫 분석 및 개선 제안.
* **문서화 봇**: 문서 생성 및 주석 추가.
* **보안 감사**: 취약점 점검.
* **테스트 생성기**: 테스트 케이스 제안.

각 에이전트는 동일한 메시지를 수신하지만 각자의 전문적인 시각에서 개별적인 답변을 제공함.

### 2. 다국어 실시간 지원

* **에이전트\_KR**: 한국어로 응답.
* **에이전트\_EN**: 영어로 응답.
* **에이전트\_JP**: 일본어로 응답.

### 3. 품질 보증(QA) 워크플로우

* **상담 에이전트**: 사용자 질문에 답변 제공.
* **QA 에이전트**: 답변의 품질을 실시간 검토하고, 문제가 발견된 경우에만 추가 의견 제시.

## 설정 방법

### 기본 구성

`openclaw.json` 파일의 최상위에 `broadcast` 섹션을 추가함 (위치는 `bindings` 섹션 옆). 키 값은 대상이 되는 WhatsApp 피어 ID(Peer ID)임:

* **그룹 대화**: 그룹 JID (예: `120363... @g.us`).
* **개인 대화(DM)**: E.164 형식의 전화번호 (예: `+8210...`).

```json
{
  "broadcast": {
    "120363403215116621@g.us": ["code-reviewer", "security-bot", "assistant3"]
  }
}
```

**동작 결과**: 해당 채팅방에서 응답 조건이 충족되면 지정된 세 에이전트가 모두 실행됨.

### 처리 전략 (Processing Strategy)

에이전트들이 메시지를 처리하는 순서를 제어함:

#### 병렬 처리 (Parallel - 기본값)

모든 에이전트가 동시에 실행을 시작함:

```json
{
  "broadcast": {
    "strategy": "parallel",
    "120363403215116621@g.us": ["agent1", "agent2"]
  }
}
```

#### 순차 처리 (Sequential)

배열에 정의된 순서대로 한 번에 하나씩 실행됨 (앞선 에이전트가 완료된 후 다음 에이전트 시작):

```json
{
  "broadcast": {
    "strategy": "sequential",
    "120363403215116621@g.us": ["agent1", "agent2"]
  }
}
```

## 동작 원리 및 세션 격리

### 메시지 흐름

1. WhatsApp 그룹에 메시지 도착.
2. **브로드캐스트 확인**: 해당 피어 ID가 `broadcast` 목록에 있는지 확인함.
3. **목록에 있는 경우**: 설정된 모든 에이전트가 실행됨. 각 에이전트는 독립적인 세션 키와 격리된 컨텍스트를 가짐.
4. **목록에 없는 경우**: 일반적인 바인딩(Bindings) 규칙에 따라 첫 번째 일치하는 에이전트만 실행됨.

### 세션 격리 상세

브로드캐스트 그룹 내의 각 에이전트는 다음 항목을 완전히 독립적으로 유지함:

* **세션 키**: 에이전트 ID가 포함된 고유 키 할당.
* **대화 이력**: 자신의 답변만 기억하며 다른 에이전트의 응답은 참조하지 않음.
* **워크스페이스 및 샌드박스**: 개별 디렉터리와 격리된 실행 환경 사용.
* **도구 및 스킬**: 에이전트별로 차별화된 권한(읽기 전용 등) 부여 가능.
* **그룹 컨텍스트 버퍼**: 최근 메시지 뭉치는 피어 단위로 공유되어 모든 에이전트가 동일한 시점의 대화 맥락을 수신함.

## 모범 사례

* **에이전트 역할 분담**: 하나의 범용 에이전트 대신, 단일 책임을 가진 여러 개의 특화 에이전트를 활용할 것.
* **이름 명확화**: 각 에이전트가 어떤 역할을 수행하는지 이름(`identity.name`)만으로도 알 수 있게 설정할 것.
* **도구 권한 최소화**: 에이전트의 역할에 따라 필요한 도구만 허용(`allow`)하여 보안을 강화할 것.
* **성능 관리**: 에이전트 수가 많아지면 응답 속도가 느려질 수 있으므로, 가급적 10개 이내로 유지하고 단순 작업에는 가벼운 모델을 사용할 것.
* **독립적 오류 처리**: 각 에이전트는 서로 영향을 주지 않으므로, 하나의 에이전트에서 오류가 발생해도 나머지 에이전트의 응답은 정상적으로 전송됨.

## 호환성 및 제한 사항

* **지원 공급자**: 현재 **WhatsApp**에서만 사용 가능함 (Telegram, Discord, Slack 지원 예정).
* **우선순위**: `broadcast` 설정은 일반 `bindings` 규칙보다 항상 우선함.
* **공유 컨텍스트 미지원**: 에이전트끼리 서로의 실시간 응답 내용을 즉시 확인하거나 조율하는 기능은 현재 지원하지 않음.
* **메시지 순서**: 병렬 처리 시 응답 메시지의 도착 순서는 보장되지 않음.
* **속도 제한**: 모든 에이전트의 발신 메시지는 해당 WhatsApp 계정의 속도 제한(Rate limit)에 합산됨.

## 문제 해결 (Troubleshooting)

* **에이전트 무응답**: 에이전트 ID가 `agents.list`에 정확히 존재하는지, 피어 ID 형식이 올바른지 확인함.
* **상세 로그 확인**: `tail -f ~/.openclaw/logs/gateway.log | grep broadcast` 명령어를 통해 실행 과정을 모니터링함.
* **한 명만 응답함**: 피어 ID가 `bindings`에는 등록되어 있으나 `broadcast` 목록에서 누락되지 않았는지 확인함.
* **성능 저하**: 샌드박스 시작 시간 및 모델 추론 속도를 점검하고, 필요 시 성능이 우수한 모델이나 가벼운 샌드박스 모드를 선택함.
