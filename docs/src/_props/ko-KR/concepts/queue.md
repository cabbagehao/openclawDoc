# 명령어 대기열 (Command Queue)

OpenClaw는 모든 채널에서 유입되는 자동 응답 실행을 인프로세스 대기열(In-process queue)을 통해 직렬화함. 이를 통해 에이전트 실행 간의 충돌을 방지하면서도, 서로 다른 세션 간에는 안전한 병렬 처리를 지원함.

## 도입 배경

* **리소스 보호**: 에이전트 실행은 LLM 호출 등 비용이 많이 드는 작업이며, 여러 메시지가 동시에 도착할 경우 시스템 리소스(세션 파일, 로그, CLI 입력 등) 경쟁이 발생할 수 있음.
* **안정성 확보**: 작업을 직렬화함으로써 업스트림 공급자의 속도 제한(Rate limit)에 걸릴 확률을 낮추고 실행 일관성을 유지함.

## 동작 방식

* **레인(Lane) 기반 관리**: FIFO(선입선출) 방식의 대기열이 각 레인별로 지정된 동시성 제한(Concurrency cap)에 따라 작업을 처리함. (기본값: 미설정 레인 1, `main` 4, `subagent` 8).
* **세션별 직렬화**: `runEmbeddedPiAgent`는 **세션 키**(레인 ID: `session:<key>`)를 기준으로 작업을 대기열에 넣음. 이를 통해 하나의 세션에서는 반드시 하나의 에이전트만 활성화되도록 보장함.
* **전역 레인 할당**: 각 세션 작업은 다시 **전역 레인**(기본값: `main`)에 할당되어, 시스템 전체의 총 병렬 실행 수가 `agents.defaults.maxConcurrent` 설정을 넘지 않도록 제어됨.
* **사용자 경험 유지**: 대기열에서 순서를 기다리는 동안에도 채널이 지원한다면 '입력 중(Typing)' 표시가 즉시 활성화되어 사용자는 끊김 없는 경험을 하게 됨. 상세 로그(Verbose) 활성화 시, 대기 시간이 2초를 초과하는 작업에 대해 안내 메시지를 출력함.

## 대기열 모드 (Queue Modes)

수신된 메시지가 현재 실행 중인 작업에 개입하거나 후속 작업을 예약하는 방식은 채널별로 설정 가능함:

* **`steer`**: 현재 실행 중인 에이전트 작업에 즉시 개입함 (다음 도구 실행 경계에서 대기 중인 호출 취소). 스트리밍 모드가 아닐 경우 `followup`으로 자동 전환됨.
* **`followup`**: 현재 작업이 완료된 후 다음 실행 순서로 대기열에 추가함.
* **`collect` (기본값)**: 대기열에 쌓인 여러 메시지를 **하나의 후속 작업**으로 합쳐서 처리함. 단, 메시지의 대상 채널이나 스레드가 다를 경우 라우팅 보존을 위해 개별적으로 처리됨.
* **`steer-backlog`**: 현재 작업에 즉시 개입하는 동시에, 해당 메시지를 후속 작업을 위해 보존함.
* **`interrupt` (레거시)**: 현재 실행 중인 작업을 즉시 강제 중단(Abort)하고 새 메시지를 실행함.

<Tip>
  **권장 사항**: 스트리밍 UI에서 중복 응답처럼 보이는 것을 방지하려면 `collect` 또는 `steer` 모드 사용을 권장함. 세션 내에서 `/queue collect` 명령어로 개별 설정하거나 `messages.queue.byChannel` 설정을 통해 채널별로 고정할 수 있음.
</Tip>

### 기본값 및 설정 예시

설정 파일에 명시되지 않은 경우 모든 채널의 기본값은 `collect`임.

```json5
{
  messages: {
    queue: {
      mode: "collect",
      debounceMs: 1000,
      cap: 20,
      drop: "summarize",
      byChannel: { discord: "collect" },
    },
  },
}
```

## 세부 옵션

이 옵션들은 `followup`, `collect`, `steer-backlog` 모드에 적용됨:

* **`debounceMs`**: 후속 작업을 시작하기 전 대기 시간 (사용자의 연속된 입력을 기다려 "잠시만요, 계속하세요"와 같은 상황 방지).
* **`cap`**: 세션당 대기열에 쌓을 수 있는 최대 메시지 수.
* **`drop`**: 용량 초과 시 처리 정책 (`old`: 오래된 것 삭제, `new`: 새 메시지 무시, `summarize`: 삭제된 내용 요약 삽입).

`summarize` 정책 사용 시, 누락된 메시지들을 불릿 목록으로 요약하여 후속 프롬프트에 자동으로 포함시킴.

## 세션별 설정 오버라이드

* 채팅창에 `/queue <mode>` 형식을 단독 메시지로 입력하여 현재 세션의 모드를 즉시 변경할 수 있음.
* 여러 옵션 조합 가능: `/queue collect debounce:2s cap:25 drop:summarize`
* `/queue default` 또는 `/queue reset`으로 설정을 초기화함.

## 적용 범위 및 기술적 특징

* Gateway 응답 파이프라인을 사용하는 모든 채널(WhatsApp, Telegram, Slack, Discord, iMessage, WebChat 등)에 적용됨.
* **`main` 레인**: 일반 수신 메시지 및 하트비트 작업이 공유하는 전역 레인임.
* **추가 레인**: `cron`, `subagent` 등 별도 레인을 운영하여 백그라운드 작업이 실시간 응답을 방해하지 않도록 설계됨.
* 순수 TypeScript 기반의 Promise 구조로 구현되었으며, 외부 의존성이나 별도의 워커 쓰레드 없이 안정적으로 동작함.

## 문제 해결 (Troubleshooting)

* 명령 처리가 멈춘 것처럼 보인다면 상세 로그를 활성화하여 `"queued for ...ms"` 메시지를 확인함으로써 대기열이 정상적으로 소비되고 있는지 점검함.
* 대기열 정체 현상이 잦다면 `agents.defaults.maxConcurrent` 수치를 조정하여 병렬 처리 용량을 늘릴 수 있음.
